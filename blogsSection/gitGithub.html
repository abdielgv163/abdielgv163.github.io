¿Qué es Git y GitHub?
Y como configurarlo

GIT y GitHub son herramientas que suelen ir de la mano y que es altamente importante de conocerlas ya que nos ayudan a establecer un ritmo de trabajo en el desarrollo de nuestros proyectos, especialmente cuando se trata de proyectos colaborativos. Debemos de tener cuidado de no confundir estas 2 herramientas, pues si bien parecen similares y van de la mano, ambas ofrecen 2 soluciones distintas que se complementan entre sí para facilitar nuestro desarrollo y alojamiento de proyectos, especialmente en programación.

GIT

GIT es el sistema de control de versiones por excelencia que nos permite trabajar fácil y rápidamente en nuestros proyectos de desarrollo tanto personales como profesionales.

Se trata de un sistema gratuito y open source que está presente en la gran mayoría de los ambientes de desarrollo profesionales. Esta herramienta suele ir acompañada de un servicio web de forja que nos permite alojar y administrar nuestros proyectos, entre los cuales destacan Github y GitLab. En este caso nos centraremos en GitHub.

¿Qué es un sistema de control de versiones?
(VCS)

Un sistema de control de versiones es un sistema que almacena los cambios que ocurren en un uno o más archivos durante el transcurso del tiempo, de manera que en algún punto seamos capaces de recuperar una versión anterior de ellos.
Este sistema nos permite revertir ciertos cambios que nos podrían estar ocasionando problemas e incluso, revertir el proyecto entero a una versión anterior. También nos permite comparar los cambios efectuados, revisar quién ha realizado las modificaciones que están ocasionando problemas, cuándo han sido introducidas, etc.

Sistema de Control de Versiones Local

El "sistema de control de versiones cotidiano" podría ser guardar el proyecto en diferentes carpetas separadas por fecha, tipo de cambios, versión del proyecto, etc. E incluso, algo que muchos hemos hecho en algún momento, es tener un archivo digamos, proyecto.txt,después guardar un proyecto2.txt , luego un proyectoFinal.txt y seguir así.
Esta forma de trabajar presenta varios inconvenientes, primero que nada estamos guardando muchos archivos que en su mayoría contienen la misma información, lo que resulta en ocupar más espacio de almacenamiento. Además, no tenemos acceso a los cambios particulares que hemos ejecutado en cada uno de los archivos y mucho menos tenemos el control sobré quién y cuándo se han introducido ciertos errores. Sin olvidar que esta forma de operar es propensa a muchos errores y que nos brinda muy poco control sobre proyectos muy grandes que contienen gran cantidad de archivos que son modificados en una misma versión, imposibilitando el control de archivos en proyectos que involucren a muchas personas.

A fin de resolver estos inconvenientes es que se comenzaron a desarrollar los sistemas de control de versiones locales que lo que contienen es una base de datos muy simple, en la que se tiene el registro de los cambios realizados a los archivos.

Un sistema de ese estilo que fue muy popular en su momento fue RCS , que aún se encuentra en muchas computadoras actuales. Decimos que esas herramientas funcionan guardando conjuntos de parches (los cambios o diferencias entre los archivos), y son capaces de recrear o "restablecer" una versión anterior a partir de estos parches.

Sistema de Control de Versiones Centralizado

Nuestro sistema de control de versiones local parece solucionar la mayor parte de los inconvenientes que teníamos, pero aún queda algo por resolver y es que, aún existe la dificultad al momento de colaborar con más personas en un proyecto (más programadores o desarrolladores). Por lo que necesitamos de un sistema que se encuentre alojado en algún lugar al que todos tengan acceso para poder percibir y manejar los cambios que se le van haciendo al proyecto.
Aquí es donde nacen los sistemas de control de versiones centralizados. Estos sistemas tienen un único servidor o servidor central, donde son almacenados nuestros archivos versionados y que también nos permiten su descarga. Facilitando el trabajo de varios desarrolladores en un mismo proyecto, permitiendo que cada quién pueda tener una copia actualizada del proyecto junto con todos sus cambios y distintas versiones.

Esto también les permite a los colaboradores tener una idea de en qué está trabajando el resto del equipo, además de que los administradores cuentan con un control detallado sobre qué puede hacer cada usuario y, es mucho más fácil administrar un sistema de control de versiones centralizado que tener que lidiar con la base local de cada cliente y colaborador.
Esta sistema no está libre de problemas, pues si llegase a ocurrir un problema con el servidor, digamos que se cae durante 1 o 2 horas; entonces estará inhabilitado durante todo este tiempo y nadie podrá continuar trabajando en el proyecto.
O incluso peor, si el disco duro en donde se tienen almacenados todos los archivos versionados se llegase a averiar y sin tener copias de seguridad disponibles, significaría comprometer gravemente el proyecto y probablemente perderlo.

Sistemas de Control de Versiones Distribuidos

Este tipo de sistemas nos brindan las soluciones que los 2 sistemas anteriores no habían logrado cumplir en su totalidad. Estos sistemas se caracterizan en que los usuarios (Clientes) no solo descargan la última copia instantánea de los archivos, sino que replica completamente el repositorio entero. De esta manera, si el servidor dejase de funcionar, cualquiera de los repositorios disponibles en las copias de los clientes pueden ser copiados al servidor con la finalidad de restaurarlo.
Prácticamente contamos con distintos clones que contienen todos los datos. Y justo GIT es un sistema de tipo Distribuido.

Muchos de estos sistemas se encargan de manejar varios repositorios remotos con los cuales pueden trabajar, de tal forma que se permite la colaboración simultanea con diferentes tipos de personas de distintas maneras dentro del mismo proyecto. Esto expande mucho los flujos de trabajo disponibles que no tenemos en los sistemas centralizados; pudiendo establecer modelos jerárquicos.

GitHub

Anteriormente hablamos sobre GIT y sobre los sistemas de control de versiones en general, y comentamos que estos sistemas suelen ir acompañados de un servicio de forja, es decir, una plataforma online que permite el desarrollo colaborativo de software, diseñada tanto para desarrollar como para difundir software; y justo eso es lo qué es GitHub.

GitHub, anteriormente conocida como Logical Awesome LLC; es una forja que nos permite alojar proyectos utilizando el sistema de control de versiones GIT. Actualmente pertenece a Microsoft, pero sigue estando enfocada en el alojamiento de proyectos Open Source.

Esta plataforma nos permite alojar proyectos en repositorios que es en donde se almacenan todos nuestros archivos versionados de nuestro proyecto. Además, permite la colaboración con mayor facilidad y brinda la posibilidad de hacer proyectos públicos o privados (aunque la tendencia es hacerlos públicos y que todos los interesados colaboren). También permite administrar el proyecto acorde a cierta "jerarquía", de manera que los administradores del proyecto puedan revisar los cambios realizados antes de ser admitidos en las versiones finales, incluso podemos almacenar los cambios en los archivos en base a la rama en la que estemos trabajando.

Repositorios

Los repositorios son lugares virtuales alojados en la nube, en donde los usuarios pueden almacenar cualquier tipo de archivo, aunque usualmente suelen ser archivos de texto plano que representan código en distintos lenguajes de programación; los cuales conforman un programa o aplicación.
Estos repositorios pueden ser públicos o privados, a elección del administrador del proyecto. Los repositorios privados nos permiten de cierta forma "ocultar" nuestros proyectos, de manera que el resto de la comunidad no los puede ver ni interactuar con ellos y esto de forma gratuita.
Pero si hay algo que hace destacar a esta plataforma es tener repositorios públicos, de manera que compartimos nuestro código mientras tenemos la posibilidad de colaborar con otros desarrolladores. Incluso, puede que nuestro proyecto le resulte interesante a determinada comunidad y que ciertas personas puedan ofrecer su apoyo para hacer algunas mejoras que nosotros siempre tendremos el control de administrar. Este es el core de los proyectos open source, compartir y permitir que la comunidad escale, finalizando con proyectos grandiosos y robustos.

Pull Request

La forma en la que otros usuarios pueden colaborar en un proyecto es a través de las Pull Request, las cuales son solicitudes de admisión de código. En otras palabras, los usuarios ajenos al proyecto envían algunas modificaciones de nuestro código, que podrían plantear una mejora de alguna función en particular; una vez recibimos esta solicitud, el administrador revisa los cambios y verifica que no provoquen ningún error en el proyecto y una vez revisado, acepta la solicitud y se incorpora al código fuente.
El o los administradores del proyecto siempre tienen la libertad de aceptar o denegar la solicitud del pull request.

GitHub Pages

GitHub también ofrece un servicio mediante el cual los usuarios pueden alojar su sitio web en los servidores de GitHub (como en el caso de esta página), sin la necesidad de un alojamiento de paga o almacenamiento extra en la nube. Estas páginas pueden ser creadas a modo de portafolios mediante un proceso muy sencillo que después explicaremos.
Además del uso básico de este servicio, es posible usar el generador de sitios web Jekyll.

Configurando GIT y GitHub

Vamos a explicar brevemente el proceso para descargar y configurar GIT en Windows, para posteriormente enlazarlo con GitHub.

Descargar GIT para Windows

Lo primero que tenemos que hacer es descargar GIT desde el sitio oficial, nos vamos a la sección de descargas y seleccionamos Windows, ya sea de 32Bits o de 64Bits; dependiendo de nuestro sistema.

Se nos descargará un instalador con un entorno gráfico muy sencillo. Vamos a realizar la instalación con normalidad. Lo único es que debemos de asegurarnos de que se nos instalará la GIT Bash, en las versiones más recientes esto ya viene por defecto; pero en caso de aparecer la opción, tendremos que marcarla.

Al menos de que ya se tenga algo de conocimiento utilizando GIT, vamos a instalar todo lo demás con normalidad con las opciones que vienen marcadas por defecto.
Una vez completamos la instalación vamos a tener dentro de nuestros programas algo llamado Git Bash y vamos a ejecutarla. Desde esta terminal es desde donde vamos a estar trabajando. Es una especie de emulación de una terminal Linux que nos permitirá trabajar más cómodamente con GIT.

Descargando GIT en Linux

Hay que tener en mente que cada distribución de Linux contiene un gestor de paquetes diferente, por lo que los comandos de instalación podrían cambiar un poco. Por ejemplo, en distribuciones como Ubuntu o Debian tenemos el gestor apt, en Arch Linux contamos con pacman, en Fedora utilizamos dnf, etc. Solo tenemos que se consientes de este pequeño cambio antes de continuar.
El comando para instalar GIT en nuestra distribución Linux es la siguiente: 


# Actualiza la lista de paquetes disponibles y sus versiones
sudo apt-get update

# Actualiza los paquetes de la lista anterior
sudo apt-get upgrade 

# Instalar GIT
sudo apt install git

# Comprobar la versión de GIT instalada
git --version

Configurar GIT

Ya sea que estemos trabajando desde Windows o Linux será el mismo proceso, pues recordemos que la Git Bash de Windows emula una consola Linux para trabajar cómodamente con GIT.

Configurando las credenciales

Las credenciales es la información que nos permite establecer conexión con GIT y GitHub. Para comenzar con todo esto, tenemos que hace lo siguiente:

    1. Crear una cuenta en GitHub
    Para ello sólo necesitamos de un correo que se vincule a nuestra cuenta y crear una contraseña como en cualquier otro sitio web. Una vez creada nuestra cuenta, vamos a copiar nuestro nombre de usuario.
    2. En la terminal de GIT
    Vamos a actualizar la información general de nuestro usuario (TENER CUIDADO CON LOS ESPACIOS): 
    
        # Actualizamos el nombre de nuestro usuarios de GitHub
        git config --global user.name "nombre_de_usuario_GitHub"
        
        # Configuramos el correo electrónico asociado a la cuenta de GitHub
        git config --global user.email "email_de_la_cuenta" 
        
        # Verificamos que la información sea correcta
        git config --list
    3. (Opcional) Cambiar la rama predeterminada de nuestro entorno local
    
        # A mí me gusta usar la rama master, por costumbre
        config --global init.defaultBranch master

        # Otra opción
        config --global init.defaultBranch main

Creación de llaves públicas y privadas para establecer la conexión (SSH)

    SSH Keygen es una herramienta para crear nuevos pares de claves de autentificación para un protocolo SSH. Permitiéndonos automatizar inicios de sesión y hosts.
    Esto nos permitirá autentificarnos en la plataforma con unas llaves que estarán ligadas a nuestro correo.
      1. Procedemos a crear nuestras llaves: 
      
        # Genera llaves SSH utilizando el protocolo de encriptación RSA 
        # Con 4096 bits de longitud
        ssh-keygen -t rsa -b 4096 -C Tu_Correo_Electronico_De_Gitub
      2. Seleccionamos la ruta en la que queremos guardar las llaves.
      Esta ruta puede ser modificada a criterio personal, pero podemos dejar la ruta predeterminada pulsando ENTER.
      3. Creación de contraseña ligada a las llaves (recomendado).
      Una vez tenemos la dirección en donde se almacenarán las llaves, podremos asignarles una contraseña para brindar mayor seguridad. Esta contraseña se ligará a nuestras llaves y se nos solicitará cada que queramos establecer conexión con GitHub.
      4. Verificamos el servidor SSH.
      Verificamos que todo se este realizando con normalidad y ejecutamos el siguiente comando: 
      
            eval "$(ssh-agent -s)"
      Y vamos a obtener una salida similar a la siguiente: 
      
            Agent pid #####
      5. Agregamos la llave privada al servidor SSH 
      
            ssh-add ~/.ssh/id_rsa

Enlazar llave pública a GitHub

    1. Ir a la sección de llaves SSH de GitHub
    Para esto vamos a entrar al sitio de GitHub, damos click en nuestro icono de imagen de perfil y nos vamos a configuración o "settings". Una vez estamos ahí, nos vamos a la sección llamada "Password and authentication". Esta sección contiene un ícono de una llave.
    Una vez estamos en esa sección vamos a ver un botón verde que diga "New SSH key" y lo vamos a presionar.
    Una vez estamos en esa sección, le asignamos un nombre a nuestra llave y volvemos a la terminal sin cerrar aún la página.
    2. Copiar la llave de autentificación desde la consola
    Una vez estamos en la consola, vamos a instalar una herramienta que nos ayudará a copiar nuestra llave de manera segura. Para ello ejecutaremos el siguiente comando: 
    
        sudo apt install xclip
    Y una vez tenemos instalada esta herramienta, ejecutamos el siguiente comando para copiar nuestra llave al portapapeles: 
    
        xclip -selection clipboard < ~/.ssh/id_rsa.pub
    En caso de querer hacerlo manual, debemos de ir a la carpeta en la que se encuentra nuestra llave. Si la mantuvimos por defecto, la ruta será ~/.ssh/id_rsa.pub.
    Una vez tenemos nuestra llave copiada, regresamos a GitHub y la pegamos en la sección de keys y mantenemos la configuración predeterminada de Authentication key.
    3. Listo! Ha quedado configurado nuestro entorno y listo para comenzar a trabajar.

Abarcando una guía rápida, pero completa de este sistema de control de versiones y su forja más popular.
